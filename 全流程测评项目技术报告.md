## 1. 项目背景与目标

本项目的目标是建立一套可复现、可审计、可扩展的 LLM 机器翻译评测流程，用于比较多个开源/本地大语言模型在缅甸语、中文、英文之间的翻译能力。评测对象覆盖通用 Chat 模型、翻译专用模型和 MoE 模型，评测方向覆盖缅中、中文缅、缅英、英文缅四个方向。

本次项目不是只跑一次脚本，而是完成了从环境拆分、本地权重组织、prompt/tokenizer 官方规范核验、smoke test、full test、BLEU/COMET 指标计算、异常结果回溯、参数修正、结果刷新到最终文档沉淀的一整套工程闭环。

核心目标如下：

- 使用项目现有 pipeline 完成正式评测，避免临时脚本造成流程不可复现。
- 优先使用本地模型权重，减少下载时间和网络不确定性。
- 在 A800 有卡环境下优先使用 `bf16`，只在显存问题明确出现时回退到 `8bit`。
- 确保 HY-MT、NiuTrans LMT、Qwen3、Gemma 系列使用符合官方文档和本地模型卡的 prompt/tokenizer/generation 设置。
- 对缅语目标方向进行正式分词，避免 BLEU 被缅语书写系统和 tokenizer 处理方式干扰。
- 使用 BLEU 和 COMET 双指标进行评估，并保留逐模型逐方向 JSON 和汇总 CSV。
- 沉淀工程经验，指导后续模型接入、流程提速、异常排查和结果解释。

项目使用两个 Conda 环境：

| 环境 | Python 路径 | 主要职责 |
| --- | --- | --- |
| `mt-infer` | `/root/miniconda3/envs/mt-infer/bin/python` | 模型加载、tokenizer、Transformers 推理、bitsandbytes 备用 |

- 一旦某个模型需要特殊 transformers 版本，可以创建额外推理环境，不影响 COMET 和整体结果汇总。
- 后续如果要引入 vLLM、SGLang 或模型专用环境，也可以按模型新增推理环境，保持评测环境稳定。

## 5. 数据集设计

### 5.1 方向与文件

测试集位于 `data/test/`，每个方向 500 行。每个方向有源文和两个参考译文。

| 方向 | 源文 | 参考 1 | 参考 2 | 目标是否缅语 |
| --- | --- | --- | --- | --- |
| `my2zh` | `data/test/my2zh/my.txt` | `data/test/my2zh/my2zh1.txt` | `data/test/my2zh/my2zh2.txt` | 否 |
| `zh2my` | `data/test/zh2my/zh.txt` | `data/test/zh2my/zh2my1_seg.txt` | `data/test/zh2my/zh2my2_seg.txt` | 是 |
| `my2en` | `data/test/my2en/my.txt` | `data/test/my2en/my2en1.txt` | `data/test/my2en/my2en2.txt` | 否 |
| `en2my` | `data/test/en2my/en.txt` | `data/test/en2my/en2my1_seg.txt` | `data/test/en2my/en2my2_seg.txt` | 是 |

注意：`zh2my` 与 `en2my` 的参考文件使用 `_seg.txt`，即已经过缅语分词处理。这是为了让 BLEU 的 hypothesis 与 reference 在相同分词粒度下比较。

### 5.2 数据长度统计

本次特别核查了缅语目标方向的参考长度，用于判断 `max_new_tokens` 是否可能导致系统性截断。

| 文件 | 行数 | 平均字符 | P95 字符 | 最大字符 | 平均分词数 | P95 分词数 | 最大分词数 |
| --- | ---: | ---: | ---: | ---: | ---: | ---: | ---: |
| `data/test/zh2my/zh2my1_seg.txt` | 500 | 162.30 | 342 | 619 | 28.95 | 59 | 107 |
| `data/test/zh2my/zh2my2_seg.txt` | 500 | 156.22 | 319 | 581 | 28.26 | 58 | 105 |
| `data/test/en2my/en2my1_seg.txt` | 500 | 162.30 | 342 | 619 | 28.95 | 59 | 107 |
| `data/test/en2my/en2my2_seg.txt` | 500 | 156.22 | 319 | 581 | 28.26 | 58 | 105 |

结论：缅语参考虽然字符长度较长，但分词后长度最大约 105-107 个空格分词 token。除 HY 外，其余模型使用 `max_new_tokens=512`，从这个统计看，不应导致全局系统性截断。

## 7. 配置体系

本项目把模型配置、prompt 模板和生成参数拆成三个文件，避免把模型逻辑散落在脚本中。

### 7.1 `configs/models.yaml`

该文件描述每个模型的元信息和推理加载策略。关键字段包括：

| 字段 | 含义 |
| --- | --- |
| `hf_id` | HuggingFace 模型 ID |
| `family` | 模型家族标签 |
| `dtype` | 默认精度，本次主要为 `bf16` |
| `default_quantization` | 默认量化方式，本次均为 `null` |
| `fallback_quantization` | 大模型回退量化方式，Gemma/Qwen 大模型设置为 `8bit` |
| `prompts` | 四个方向对应的 prompt template key |
| `load_mode` | 加载方式，目前为 `causal_lm` |
| `trust_remote_code` | 是否启用 remote code |
| `attn_implementation` | attention 实现，可留空 |
| `device_map` | 设备映射，本次为 `auto` |
| `tokenizer_mode` | tokenizer 模式，本次多为 `chat` |
| `uses_chat_template` | 是否调用 `apply_chat_template` |
| `chat_add_generation_prompt` | chat template 是否追加 assistant generation prompt |
| `padding_side` | padding 方向，大模型批量推理中常用 `left` |
| `max_input_length` | 输入截断上限 |
| `stop_strings` | 后处理截断字符串，例如 Gemma 的 `<end_of_turn>` |
| `tokenizer_kwargs` | 传入 tokenizer 的额外参数 |
| `apply_chat_template_kwargs` | 传入 `apply_chat_template` 的额外参数，如 `enable_thinking=false` |

模型特异配置示例：

- HY-MT：`chat_add_generation_prompt: false`，这是官方要求。
- NiuTrans：`tokenizer_kwargs.fix_mistral_regex: true`，用于规避 tokenizer regex 兼容问题。
- Qwen3/Qwen3-MoE：`apply_chat_template_kwargs.enable_thinking: false`。
- Gemma3/Gemma4/Gemma4-MoE：`apply_chat_template_kwargs.enable_thinking: false`，Gemma 系列还设置了 `<end_of_turn>` stop string。

### 7.2 `configs/prompts.yaml`

该文件保存所有 prompt 模板。不同模型可以绑定不同模板。

HY-MT 官方风格：

```text
将以下文本翻译为中文，注意只需要输出翻译后的结果，不要额外解释：

{source}
```

NiuTrans 官方语言标签风格：

```text
Translate the following text from Burmese into Chinese:
Burmese: {source}
Chinese:
```

通用 Chat 模型风格：

```text
You are a translation engine.
Translate the following Myanmar text into Chinese.
Output only the translation.

{source}
```

工程经验：小模型和翻译专用模型对 prompt 模板非常敏感。HY 和 NiuTrans 的早期异常结果说明，不能假设所有 Chat 模型都适合统一的 system/user prompt。正式评测必须先读官方文档和本地 `tokenizer_config.json`。

### 7.3 `configs/generation.yaml`

生成参数分为 defaults 和 model-level overrides。当前关键参数：

| 模型 | max_new_tokens | decoding 参数 |
| --- | ---: | --- |
| `hy_mt1_5_7b` | 2048 | `do_sample=true,temperature=0.7,top_p=0.6,top_k=20,repetition_penalty=1.05` |
| `niutrans_lmt_60_8b` | 512 | `num_beams=5,do_sample=false` |
| `gemma3_27b_it` | 512 | `do_sample=true,temperature=1.0,top_p=0.95,top_k=64` |
| `gemma4_31b_it` | 512 | `do_sample=true,temperature=1.0,top_p=0.95,top_k=64` |
| `gemma4_moe_26b_a4b_it` | 512 | `do_sample=true,temperature=1.0,top_p=0.95,top_k=64` |
| `qwen3_32b` | 512 | `do_sample=true,temperature=0.7,top_p=0.8,top_k=20,min_p=0.0` |
| `qwen3_moe_30b_a3b` | 512 | `do_sample=true,temperature=0.7,top_p=0.8,top_k=20,min_p=0.0` |

参数选择依据：

- HY-MT：按 HY-MT 官方模型卡示例，尤其是 `add_generation_prompt=false` 与 `max_new_tokens=2048`。

- Gemma：按本地/官方 generation config 使用 `temperature=1.0,top_p=0.95,top_k=64`。
- `max_new_tokens=512`：对当前 500 行测试集足够覆盖缅语目标长度；HY 官方给出 2048，因此单独保留。

## 8. Pipeline 技术路线

### 8.1 总体流程

项目优先使用 `scripts/run_pipeline.py` 统一编排。pipeline 的 stage 顺序如下：

1. `download_models.py`
2. `translate_batch.py`
3. `segment_myanmar.py`
4. `validate_outputs.py`
5. `eval_bleu.py`
6. `eval_comet.py`
7. `summarize_metrics.py`

Smoke 与 full 共用同一套 pipeline：

- Smoke 模式会在 `outputs/smoke/smoke_inputs/` 创建截断后的源文和参考，确保翻译、validate、BLEU、COMET 都在相同行数上运行。
- Full 模式直接使用 `data/test/` 下完整 500 行数据。

### 8.2 执行策略

本次执行坚持单模型顺序运行：

1. 先按模型跑 smoke。
2. smoke 满足模型加载、双方向/四方向产出、缅语分词、validate、BLEU、COMET 后，再跑 full。
3. full 也按模型顺序执行，不一次性并发多个大模型。
4. 每个模型完成后检查输出文件、BLEU JSON、COMET JSON 和 summary。

这种策略牺牲了一些并行速度，但极大降低了定位成本。对于 A800 单卡，多个 30B 级模型并发几乎没有意义，反而容易造成 OOM、缓存污染和日志混乱。

### 8.4 离线缓存策略

`mt_eval/pipeline.py` 在执行 subprocess 时设置：

```text
HF_HOME=/root/autodl-tmp/root-cache/huggingface
HUGGINGFACE_HUB_CACHE=/root/autodl-tmp/root-cache/huggingface/hub
TRANSFORMERS_CACHE=/root/autodl-tmp/root-cache/huggingface/hub
HF_HUB_OFFLINE=1
```

这样做的目的：

- 正式评测优先使用本地权重和本地 COMET 缓存。
- 避免运行中阻塞在网络下载。
- 避免同一个模型因远端文件变更产生不可控差异。
- 明确暴露本地权重缺失问题，而不是静默下载或半下载。

工程经验：在 AutoDL 这类按时计费环境中，权重下载要尽量放在无卡阶段或单独准备阶段；有卡阶段只做必要核验和推理。

## 9. 推理后端实现细节

### 9.1 后端选择

当前使用 HuggingFace Transformers：

```python
AutoTokenizer.from_pretrained(...)
AutoModelForCausalLM.from_pretrained(...)
model.generate(...)
```

未使用 vLLM。选择 Transformers 的原因：

- 本项目首先追求结果有效性和官方模板一致性。
- HY-MT、NiuTrans、Gemma、Qwen 的 chat template、generation config、tokenizer kwargs 可以直接按官方示例落地。
- vLLM 可能带来 tokenizer、chat template、stop token、thinking 解析、量化加载差异，不应和 prompt 修复混在同一轮正式实验中。

后续如果追求速度，应新增独立 vLLM backend，并通过 smoke 等价性测试验证输出结构和指标不会系统性偏移。

### 9.2 模型加载

模型加载逻辑位于 `mt_eval/inference.py`：

- `dtype=bf16` 时传入 `torch_dtype=torch.bfloat16`。
- `device_map=auto` 交给 accelerate/transformers 自动映射。
- `quantization=8bit` 时使用 `BitsAndBytesConfig(load_in_8bit=True)`。
- `4bit` 在 CLI 中被显式拒绝，避免无意中使用低精度结果。

本次没有触发 `8bit` 回退，但代码路径保留给大模型 OOM 场景。

### 9.3 Chat template

如果 `uses_chat_template=true`，推理输入构造为：

```python
messages = [[{"role": "user", "content": prompt}] for prompt in prompts]
tokenizer.apply_chat_template(
    messages,
    add_generation_prompt=settings.chat_add_generation_prompt,
    tokenize=True,
    return_tensors="pt",
    padding=True,
    truncation=settings.max_input_length is not None,
    max_length=settings.max_input_length,
    return_dict=True,
    **settings.apply_chat_template_kwargs,
)
```

关键点：

- HY-MT：`add_generation_prompt=false`。
- NiuTrans、Gemma、Qwen：默认 `add_generation_prompt=true`。
- Qwen/Gemma：通过 `apply_chat_template_kwargs` 显式设置 `enable_thinking=false`。
- 所有模型只构造 user message，不额外注入 system prompt，避免污染模型官方模板。

### 9.4 批量与续跑

pipeline 默认 `batch_size=10`，并传入 `--resume`。续跑逻辑：

- 如果输出文件存在且以换行结尾，统计已完成行数。
- 从未完成行继续翻译。
- 如果输出行数超过源文行数，直接报错。
- 如果输出文件不是以换行结尾，认为 resume 状态无效，避免半行污染。

工程经验：长时间推理必须支持 resume。否则一旦 30B 模型中途失败，重跑成本很高。

### 9.5 输出后处理

输出后处理做两件事：

- 多行输出归一化为单行。
- 如果配置了 `stop_strings`，遇到 stop string 截断。

这样可以保证每个源句对应输出文件中的一行，方便 validate、BLEU、COMET 对齐。

## 10. 模型专属问题与修复经验

### 10.1 HY-MT

官方依据：

- `https://github.com/Tencent-Hunyuan/HY-MT`
- `models/hy_mt1_5_7b/README.md`

关键官方要求：

- 使用 `apply_chat_template`。
- `add_generation_prompt=False`。
- 不默认添加 system prompt。
- 生成上限示例为 `max_new_tokens=2048`。
- 推荐采样参数包括 `top_k=20`、`top_p=0.6`、`temperature=0.7`。

本项目修复：

- 配置 `chat_add_generation_prompt: false`。
- HY prompt 切换为官方风格。
- generation 参数切换为官方推荐。
- 旧异常结果覆盖重跑。

经验：HY 这类翻译专用小模型对模板更敏感，手写通用 Chat prompt 会导致输出乱码、解释性文本、复读或指令遵循失败。

### 10.2 NiuTrans LMT

官方依据：

- `https://github.com/NiuTrans/LMT`
- `models/niutrans_lmt_60_8b/README.md`

关键官方要求：

- 使用语言标签 prompt。
- 使用 `apply_chat_template(..., add_generation_prompt=True)`。
- `max_new_tokens=512`。
- `num_beams=5`。
- `do_sample=False`。

本项目修复：

- prompt 改为 `Burmese: ... Chinese:`、`Chinese: ... Burmese:` 等官方标签格式。
- tokenizer 加载增加 `fix_mistral_regex=true`。
- 使用 beam search，而不是通用 sampling。

经验：NiuTrans 的 `my2zh/my2en` 在修复后恢复正常量级，证明早期异常主要是流程问题。`zh2my/en2my` 仍低，且 BLEU brevity ratio 偏低，应继续做样例级错误分析。

### 10.3 Qwen3 与 Qwen3-MoE

关键要求：

- Qwen3 默认具有 thinking 能力。
- 翻译评测不应启用 thinking。
- 官方建议非 thinking 模式使用 `temperature=0.7,top_p=0.8,top_k=20,min_p=0`。

本项目设置：

- `enable_thinking=false`
- `temperature=0.7`
- `top_p=0.8`
- `top_k=20`
- `min_p=0.0`
- `do_sample=true`

经验：对 reasoning-capable 模型，thinking 开关必须显式记录。否则输出中可能出现 `<think>` 或隐式推理内容，污染翻译输出和评测指标。

### 10.4 Gemma3、Gemma4 与 Gemma4-MoE

关键要求：

- 使用模型自带 chat template。
- Gemma4 模型卡示例支持 `enable_thinking=false`。
- generation config 推荐 `temperature=1.0,top_p=0.95,top_k=64`。

本项目设置：

- `enable_thinking=false`
- `temperature=1.0`
- `top_p=0.95`
- `top_k=64`
- `do_sample=true`
- `max_new_tokens=512`
- stop string 包含 `<end_of_turn>`

经验：Gemma4-31B 和 Gemma4-MoE 在缅语目标方向表现明显领先，说明大模型的多语泛化能力对缅语生成质量有帮助。

## 11. 评测指标实现

### 11.1 BLEU

BLEU 使用 `sacrebleu.corpus_bleu`，不同方向使用不同 tokenizer：

| 方向 | sacreBLEU tokenizer |
| --- | --- |
| `my2zh` | `zh` |
| `zh2my` | `none` |
| `my2en` | `13a` |
| `en2my` | `none` |

为什么缅语方向使用 `none`：

- hypothesis 和 reference 都先通过外部分词处理。
- 如果 sacreBLEU 再做额外 tokenizer，可能破坏缅语分词结果。
- 因此 `zh2my/en2my` 的 BLEU 前提是 hypothesis 和 reference 都在同一分词粒度上。

### 11.2 缅语分词

缅语目标方向需要额外执行：

```bash
python scripts/segment_myanmar.py \
  --backend pyidaungsu \
  --input outputs/full/translations/raw/<model>_zh2my.txt \
  --output outputs/full/translations/bleu_ready/<model>_zh2my.seg.txt
```

正式 backend 是 `pyidaungsu`。项目也保留 `command` 和 `fallback`：

- `pyidaungsu`：正式默认。
- `command`：允许接入外部正式分词器。
- `fallback`：仅用于本地 smoke，不推荐正式结果使用。

### 11.3 COMET

COMET 使用：

- 模型：`Unbabel/wmt22-comet-da`
- 本地缓存路径：`root-cache/huggingface/hub/models--Unbabel--wmt22-comet-da/...`
- 每个方向使用两个参考。
- 分别对 ref1/ref2 计算 COMET。
- `comet_mean = (comet_ref1 + comet_ref2) / 2`，保留 10 位小数。
- 当前 `model.predict(..., batch_size=1, gpus=0)`，即 CPU 评估。
- 没有额外归一化，COMET 输出本身通常处于接近 0-1 的范围，但本项目未自行缩放。

工程经验：COMET CPU 阶段比 BLEU 慢很多，是后续优化的重要目标。可以给 `eval_comet.py` 增加 `--gpus` 和 `--batch-size`，但这属于评测加速改造，不应和模型结果修复混在同一实验中。

## 12. smoke test 判定标准

本项目中 smoke 不是只跑 help 或只生成一两句，而是必须覆盖完整链路：

1. 模型能成功加载。
2. 四个方向能产出文本。
3. `zh2my/en2my` 能完成缅语分词。
4. `validate_outputs.py` 通过行数和空行检查。
5. BLEU JSON 能写出。
6. COMET JSON 能写出。
7. summary CSV 能更新。

这个标准很重要。只验证模型能 `generate` 不够，因为正式评测常见失败点包括：

- 输出行数与源文不一致。
- 缅语分词器不可用。
- COMET 模型缺失或缓存路径错误。
- BLEU reference 路径错误。
- Chat template 导致输出为空或输出解释性文本。

## 15. 异常排查与修复过程中的关键发现

### 15.1 HY 和 NiuTrans 早期异常不是模型能力结论

早期 HY 和 NiuTrans 的结果很反常。经过核查，核心问题不是模型本身，而是调用流程未完全符合官方推荐：

- HY 需要 `add_generation_prompt=false`，不能直接套通用 assistant prompt。
- NiuTrans 需要语言标签格式 prompt 和 beam search。
- tokenizer 兼容细节如 `fix_mistral_regex` 会影响稳定性。

工程经验：当小模型或翻译专用模型表现反常时，第一步不是下模型能力结论，而是检查：

- `tokenizer_config.json`
- `generation_config.json`
- 官方 README 的推理示例
- 是否使用 `apply_chat_template`
- `add_generation_prompt` 是否正确
- 是否引入了不该有的 system prompt
- 是否打开了 thinking
- 是否使用了错误 decoding 参数

### 15.2 thinking 模式必须显式控制

Qwen3 和部分 Gemma 模型具备 reasoning/thinking 能力。翻译评测中如果启用 thinking，可能出现：

- 输出包含 `<think>`。
- 输出包含解释而非译文。
- 输出长度增加，影响 BLEU 和 COMET。
- 不同模型之间比较不公平。

本次所有 Qwen/Gemma thinking-capable 模型均显式设置 `enable_thinking=false`，并在结果中未检出 thinking 标记。

### 15.3 `max_new_tokens` 不是全局低分主因

用户提出 `zh2my/en2my` 低分是否可能由缅语天然较长导致 `max_new_tokens` 截断。本次做了三层核查：

1. 查看 `configs/generation.yaml`：HY 为 2048，其余为 512。
2. 统计缅语参考分词长度：最大约 105-107。
3. 查看 BLEU signature 中的 brevity penalty 和 ratio。

结果：

- Gemma4、Gemma4-MoE、HY、Qwen3 等 Myanmar-target 输出 ratio 接近或高于 1。
- NiuTrans `zh2my` ratio=0.689，`en2my` ratio=0.844，确实输出偏短。
- 但由于 512 明显高于参考分词长度，NiuTrans 偏短不应简单归因于全局生成上限。

后续应检查 NiuTrans 的 EOS、beam search 早停、样例输出质量和 prompt 是否需要更精细的语言标签。

### 15.4 BLEU 和 COMET 的解释不能混同

BLEU 更偏字符串匹配和 n-gram overlap；COMET 更偏语义质量估计。缅语目标方向中可能出现：

- BLEU 低但 COMET 高：模型语义可能接近，但分词、表达方式、参考差异导致 n-gram 匹配低。
- BLEU 高且 COMET 高：更可靠的强表现。
- BLEU 低且 COMET 低：更可能是真实质量问题。

因此正式报告应同时保留 `main_results.csv` 和 `detail_results.csv`，不要只看单一指标。

## 18. 8bit 回退策略

本次未使用 `8bit`，但项目保留了回退路径。适用条件：

- CUDA OOM。
- 模型加载失败且明显与显存相关。
- `bf16` 运行明显不稳定。

原则：

- 只对失败模型切换 `8bit`。
- 其他模型保持原流程和原精度。
- 不使用 `4bit`，除非 `8bit` 明确不可行且用户确认。
- 切换精度必须在最终报告中明确标注。

示例：

```bash
/root/miniconda3/envs/mt-infer/bin/python scripts/translate_batch.py \
  --input data/test/zh2my/zh.txt \
  --output outputs/full/translations/raw/qwen3_32b_zh2my.txt \
  --model-name qwen3_32b \
  --model-path models/qwen3_32b \
  --direction zh2my \
  --config-root configs \
  --batch-size 10 \
  --resume \
  --quantization 8bit
```

工程经验：量化会引入新的变量。除非显存无法承受，否则正式性能比较应优先保持 `bf16`。

## 19. 工程收获

### 19.1 评测有效性优先于跑得快

一开始就切 vLLM 或并行跑多个模型，可能更快，但也会让问题定位复杂化。对于这类多模型、多语种、多指标项目，应先建立一个慢但可信的基准流程，再考虑加速。

本项目最终选择：

- 单模型顺序执行。
- 先 smoke 再 full。
- 先修官方模板，再讨论后端提速。
- 所有结果写入标准目录。

这是正确的工程优先级。

### 19.2 Prompt/template 是翻译专用模型的核心变量

HY 和 NiuTrans 的异常说明，prompt 不是简单文案，而是模型接口协议的一部分。尤其是小模型和翻译专用模型，官方模板之外的格式可能直接让模型失效。

未来接入新模型时，必须先做以下检查：

- 官方 README。
- 本地 `README.md`。
- `tokenizer_config.json` 中的 `chat_template`。
- `generation_config.json`。
- 官方示例中的 `add_generation_prompt`。
- 是否需要 system prompt。
- 是否需要语言标签。
- 是否需要特殊 stop token。

### 19.3 评测环境要可拆可换

本次两个环境足够完成任务，但未来可能出现：

- 某个模型需要较新 transformers。
- 某个模型需要较旧 tokenizer。
- vLLM 需要单独 CUDA/torch 组合。
- COMET 升级影响 torchmetrics/lightning。

因此推荐策略是：

- `mt-eval` 保持稳定。
- `mt-infer` 作为默认推理环境。
- 对不兼容模型创建 `mt-infer-<model-family>`。
- 每个结果记录实际 Python 路径和关键包版本。

### 19.4 结果解释需要同时看指标和样例

BLEU/COMET 可以发现异常，但不能完全解释原因。NiuTrans `zh2my/en2my` 是典型例子：

- BLEU 低。
- COMET 也相对低。
- brevity ratio 显示输出偏短。
- `max_new_tokens` 又不像直接原因。

下一步必须进入样例级分析：

- raw 输出是否完整。
- 是否提前 EOS。
- 是否缺少句尾。
- 是否语言混杂。
- 是否逐字翻译。
- 是否分词器影响。
- 是否两个参考之间差异过大。

### 19.5 文档和日志是正式评测的一部分

本次保留了：

- pipeline 日志。
- full 输出。
- BLEU JSON。
- COMET JSON。
- summary CSV。
- `反馈报告.md`。
- `outputs/full/EVALUATION_SUMMARY.md`。
- 本技术报告。

这使得后续任何人都可以追踪：

- 使用了哪个模型目录。
- 使用了哪个 Python。
- 使用了什么 prompt。
- 使用了什么 generation 参数。
- 结果是否覆盖所有方向。
- 异常是如何修复的。

## 20. 后续工作建议

### 20.1 新增 vLLM backend，但不要覆盖基准

建议新增可选后端：

- `--backend transformers`
- `--backend vllm`

并建立等价性 smoke：

- 同一模型。
- 同一 prompt。
- 同一 generation 参数。
- 同一测试子集。
- 比较输出行数、空行、语言、特殊标记和指标差异。

只有通过等价性验证后，才把 vLLM 用于大规模 full。

### 20.2 COMET GPU 化

当前 COMET 固定 CPU：

```python
model.predict(data, batch_size=1, gpus=0)
```

建议增加：

- `--batch-size`
- `--gpus`
- `--num-workers`

这样可以显著缩短评测时间。但修改后要验证 COMET 分数是否一致或只存在浮点级差异。

### 20.3 建立样例级错误分析报告

建议为每个低分模型方向生成抽样表：

- source
- hypothesis raw
- hypothesis segmented
- reference 1
- reference 2
- sentence-level BLEU 或 chrF
- sentence-level COMET
- 输出长度 ratio
- 是否包含源文复制
- 是否空输出
- 是否语言不匹配

优先分析：

- `niutrans_lmt_60_8b zh2my`
- `niutrans_lmt_60_8b en2my`
- `qwen3_moe_30b_a3b zh2my`
- `qwen3_moe_30b_a3b en2my`

### 20.4 强化运行元数据记录

建议每次 full run 自动记录：

- 日期时间。
- GPU 型号和驱动。
- Python 路径。
- 包版本。
- 模型目录。
- 模型文件 hash 或 snapshot revision。
- config 文件 hash。
- pipeline 命令。
- 是否 offline。
- 是否使用量化。

当前 `translate_batch.py` 支持 `--log-json`，后续可让 pipeline 默认为每个模型方向写入 log JSON。

### 20.5 扩展指标

可考虑增加：

- chrF：对形态丰富语言和分词不稳定语言更稳健。
- COMETKiwi 或 reference-free QE：用于参考质量有限场景。
- Language ID：检测输出语言是否正确。
- copy rate：检测源文复制。
- length ratio：检测过短或过长。
- empty rate：检测空输出。
- special token rate：检测 `<think>`、`<end_of_turn>` 等污染。

## 21. 未来项目推荐标准流程

以后做类似 LLM 翻译评测，建议按以下顺序执行：

1. 收集模型官方文档和本地模型卡。
2. 检查 `tokenizer_config.json` 和 `generation_config.json`。
3. 为每个模型写入明确的 `models.yaml`、`prompts.yaml`、`generation.yaml`。
4. 无卡阶段下载权重和 COMET 缓存。
5. 有卡阶段先跑 `nvidia-smi` 和 Python import checks。
6. 按单模型顺序跑 smoke。
7. smoke 通过后按单模型顺序跑 full。
8. 每个模型完成后检查 raw、segmented、BLEU、COMET、summary。
9. 异常结果先查模板、tokenizer、thinking、长度、语言、空行。
10. 修复后只重跑受影响模型方向，并记录原因。
11. 完成后重新生成 summary 和报告。
12. 最后运行测试和覆盖校验。

## 23. 一句话总结

本项目最大的工程收获是：LLM 翻译评测的可信度不只取决于模型和指标，更取决于是否严格遵守每个模型的官方调用协议、是否把 prompt/tokenizer/thinking/decoding 当作实验变量管理、是否有完整的 smoke-full-validate-metrics-summary 闭环，以及是否在异常结果出现时先排查流程有效性再解释模型能力。
